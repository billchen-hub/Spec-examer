這個系統做什麼（一頁版）

把 spec 文件丟進去 → AI 出題 → 另一個 AI 作答 → 再一個 AI 當裁判評分 → 產出報告與改善建議。

為什麼拆三個角色

  • Examiner 產出題目 — 可用 Claude Code（免 API 費）或 Nexus AI（自動化）
  • Examinee 是要被評測的目標（地端 AI，自帶 RAG）
  • Judge 獨立於 Examinee 運作，避免「球員兼裁判」造成分數虛高

為什麼出題與考試流程是分開的

  • 題庫是 JSON 檔，可以重用、手動編輯、跨 session 累積
  • 答題和評分可拆到不同機器跑（--answer-only 模式）
  • Prompt 都走 YAML 模板 — 改出題/評分規則不用碰 Python

三角色架構

Examiner
Claude Code / Nexus AI
讀 spec → 產題庫 JSON。
輸入：檔案或資料夾（可混合）
副檔名：.pdf / .md / .txt / .log
Examinee
Nexus AI (share_code)
被評測對象。
接題目 → 回答案。
不帶 spec 給它 — 它有自己的 RAG。
Judge
Nexus AI (另一個 share_code)
比對參考答案與實際答案。
給 0-100 分 + feedback + 改善建議。
最後再產出 overall suggestion。
關鍵：三個角色只透過 JSON 題庫 / answers.json 互通；彼此不共享 prompt、不共享狀態。要換任何一方（包括換到 Hermes）都是獨立工作。

CLI 三種模式

python exam_runner.py [...]
Flag流程輸出用途
--generate PATH ... run_generate() question_bank/*.json 自動出題（Nexus）
--answer-only run_answer_only() results/answers_*.json 只答題，帶回 Claude Code 評分

為什麼有 --answer-only：Nexus API 只能在公司網走；評分用 Claude Code 不花 API 額度 — 拆成兩階段就是為了這個場景。

資料流（end-to-end）

出題：--generate

paths → collect_spec_files()    (dedup by realpath)
     → read_spec_file() × N    (pdf / text)
     → combined = "--- FILE: a.md ---\n...\n\n--- FILE: b.md ---\n..."
     → examiner_prompt + combined → NexusClient
     → _parse_questions_response() (multi-strategy regex)
     → question_bank/<stem>_<N>q_<ts>.json

考試：預設 (run_exam)

load bank → select_questions()   (full | random)
  for each q:
    → examinee prompt  → NexusClient
    → persist answers.json after every question
    → judge.score_answer()  → NexusClient
  → judge.generate_overall_suggestion()
  → ReportGenerator.generate_all() (html / json / md)
每題即時落盤 answers.json 是刻意的 — Nexus 偶爾 timeout 或公司網路掉線，落盤後可以用 --answer-only 續跑或手動補評分。

spec_loader.py 設計重點

2026-04-22 新加的模組（commit 96d4f04），是 --generate 能吃多檔/資料夾/PDF/MD 混合輸入的核心。

1. 去重用 realpath 不是 abspath

def _add(path):
    key = os.path.realpath(path)   # 不是 abspath
    if key in seen: return
為什麼：Windows junction / symlink 下，同檔不同路徑 abspath 不同但 realpath 相同。Code review 抓過的 major issue。

2. _SKIP_DIRS 只在遞迴子層生效

使用者把 tmp/ 當輸入 → 裡面的 .git 跳過。但直接把 .git 當輸入 → 照掃（尊重使用者意圖）。這個邊界有測試保護 TestSkipDirsBoundary。

3. 全空內容要炸

if non_empty == 0:
    raise SpecLoadError("All N spec file(s) produced no extractable text ...")
為什麼：掃描版 PDF（沒 OCR）pypdf 回空。全空就拒絕送 prompt — 避免浪費 API 產垃圾。個別檔案空只 warning，不阻斷。

Prompt 模板機制

結構

name: "..."
description: "..."
template: |
  ...{{variable}}...

替換規則

  • regex \{\{(\s*\w+\s*)\}\}
  • 找不到 key 的 placeholder 原樣保留（不會 KeyError）
  • 只支援簡單 key — 不支援 {{a.b}}
  • 不是 jinja2 — 不支援 {% if %}

加新角色的流程

  1. 新增 prompts/<role>.yaml
  2. 上游呼叫 prompt_loader.render("<role>", {...})
  3. 不用動 prompt_loader.py
為什麼不用 jinja2：prompt 的邏輯應該用自然語言寫，不是模板分支。極簡是刻意的。要條件分支時再升級。

三份 prompt 各管什麼

examiner.yaml題型比例、難度分佈、JSON 輸出 schema
examinee.yaml答題口吻、要求引用章節編號
judge.yaml評分權重（50/30/20）、JSON 輸出 schema；另含 overall_template

Nexus → Hermes 遷移路徑

公司未來要切 Hermes agent + Qwen 3.5。不用重寫 — 就改一個檔：nexus_client.py

實際被呼叫的 method 只有一個

def generate_response_sync(
    self, share_code, history, system_prompt=None, files=[]
) -> str

遷移 checklist

  1. 建 hermes_client.py 或直接改 nexus_client.py
  2. 保留 method 簽章（特別是回傳純字串、不是 dict）
  3. 錯誤分支要回 "[ERROR] ..." 開頭 — 上游用 startswith 判斷
  4. Timeout handling 要保留（目前 sync = 120s）
  5. Payload：現在是 {shareCode, prompt, previousMessage, files}；Hermes 若走 OpenAI-compat 就改成 {messages: [...]}
  6. Config.yaml 的 credentials 結構跟著調
為什麼現在不先抽 LLMClient interface：只有一家供應商時，抽象會讓讀者多跳一層找實作。等要接第二家時再 extract — YAGNI。

已知限制 / Debug Tips

Windows cmd.exe 編碼雷
.bat / .cmd / requirements.txt 一律純 ASCII。pip 在 Windows 用 cp950 讀 requirements.txt，UTF-8 非 ASCII 字元會 crash。
.bat if block 變數
要用 setlocal EnableDelayedExpansion + !var!，不是 %var%。Code review 抓過的 blocker。
掃描版 PDF 沒文字
pypdf 只抽 embedded text — 掃描版 PDF 會回空。解法：請使用者先 OCR 轉成 MD。
（在 spec_loader 加 OCR 可行，但會帶入 Poppler / Tesseract 外部相依，目前不做。）
Nexus 回應 JSON 不穩
LLM 有機率回 markdown fence / 多段解釋 / 少大括號。多策略 regex 抽取：出題失敗 → 原始 response 存到 raw_response_*.txt；評分失敗 → score=0，不中斷整場考試。
中途掉線可以續
answers.json 每題即時落盤。看 completed_questions 知道斷在哪 → 手動 slice 題庫 → --answer-only 續跑 → 合併後交 Claude Code 評分。
