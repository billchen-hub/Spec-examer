TL;DR

用 AI 考 AI 對
規格文件的理解程度。

把公司的 spec 文件當作教材：由 Nexus 地端 AI 出題（公開文件可改用 Claude Code），Nexus 地端 AI 答題，再由 Nexus 當裁判評分。 產出 HTML / JSON / suggestions.md 三份報告，量化地端 LLM 對技術規格的掌握度，並指出待補強的盲點。

目的：在地端模型上線、模型升級、或 RAG 改版時，提供可重現的迴歸測試， 而不是「感覺好像比較準」。

三角色架構

把「考試」這件事拆成
出題 / 答題 / 評分三個獨立角色。

每個角色用不同的 AI、可獨立替換、可獨立評估。不混在同一個 prompt 裡是這個系統的核心設計。

ROLE 01

EXAMINER

出題者

讀 spec 文件，依模板產出題庫 JSON。依文件機密等級選擇出題者：內部 / 機密文件走地端 AI，公開文件可用 Claude Code。

By 主：Nexus 地端 AI
輔：Claude Code
ROLE 02

EXAMINEE

答題者

受測對象。接到題目給答案，不另外提供 spec，仰賴自身 RAG 與訓練。

By Nexus 地端 AI（公司內 LLM）
ROLE 03

JUDGE

裁判

對照題目與答案，依 rubric 評分並寫改善建議。產出 suggestions.md。

By Nexus 地端 AI（同模型或不同模型）

ROLE 03 / 裁判

裁判：依 rubric 評分，
不只給分數，還寫建議。

JUDGE

裁判

把「題目 + 標準答案 + 受測者答案」一起餵給 Judge，套用 prompts/judge.yaml 的評分標準（rubric），輸出分數 + 評語 + 改善建議。

建議會彙整成 suggestions.md，標出系統性弱點 — 例如「對 ECC 章節常答錯」、「常把單位搞混」— 便於改 RAG 或微調訓練資料。

第二件事：換成你們的 LLM、你們的 spec、你們的標準。

換 LLM 後端（最常見的客製） 把 nexus_client.py 換成你們公司的 client。只要保留 generate_response_sync(prompt, **kwargs) → str 這個介面，其餘程式不用動。未來 Hermes 上線同樣套路。

06
客製出題模板 改題型分布、難度、語言、題數上限。YAML 即可，不用碰程式。{{spec_text}} 等變數由 prompt_loader 自動代入。

07

Edit prompts/judge.yaml rubric · weights · output_schema
08
放自己的 spec、改報告外觀 specs/ 內可放任意 PDF/MD/TXT/LOG 與資料夾。報告樣式（公司 logo、配色、欄位）改 report_generator 的 Jinja2 模板。

CAVEATS & CONTACT

已知限制與起手式。
踩雷前先讀。

已知限制

  • 純圖片 PDF（無文字層）會被 spec_loader 跳過並記 warning — 需另接 OCR。
  • Nexus 偶爾回傳非嚴格 JSON（多餘逗號、單引號）— judge.py 有 best-effort fallback，但建議監控失敗率。
  • Windows cmd 的中文編碼坑：.bat 內不要寫中文，註解一律英文（cp950 vs UTF-8）。
  • 考試中斷可用 --resume 接續；但若題庫變了，必須重跑整批。
  • 30 個測試不 mock 網路，跑單元測試不會打 Nexus；e2e 要自己手動跑。

給接手同事的起手式

  • 讀 HANDOVER.md（工程細節）+ 本 deck（全貌）。
  • Day 1：環境跑通 + sample spec e2e。
  • Week 1：替換 nexus_client.py、調 prompts。
  • Week 2：放真實 spec、跑第一輪 baseline、留存報告做後續比對基準。
  • 更新 CLAUDE.md（專案根）成你們團隊的 context — Claude Code 會自動讀。
